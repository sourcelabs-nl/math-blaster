# Competition Rules

The rules and judging criteria for the game-jam competition this project is built for.

## TL;DR

Make a (theme to be communicated) game. Winner gets **100 euros, cash**.

## Requirements

- Make a **retro styled** game.
- Teams are allowed but not necessary.
- It **must be educational**.
- You must **host your game on a webserver** that people in the office can connect to and play.
- The game must be **playable in a browser**.
- The game must use **only local storage and no backend**.

## Judging criteria

Your game will be judged on:

- **Fun** — Is it actually fun to play?
- **Music and sound effect usage** — Self explanatory.
- **Educational value** — Could you convince my dad that this game has "educational value"?
  Not too strict here, but games that are genuinely fun AND actually educational score highly.
- **Beauty** — Does it look dope? Does it look retro (80's / 90's / early 00's)?
- **Bonus: personal connection** — If the subject matter is personally important to you, you
  get extra points (not literal points, but you get the idea).

## Advice

Work incrementally. Use commit tags as checkpoints where you know the game works (a functional
main branch with feature branches, or however you prefer). Presenting a game that actually
works is crucial to winning.

## Time limit

At finish time all laptops are closed and no further work is permitted. This includes running
agentic tasks: no using Claude mobile with the laptop cracked open. Doing so means
disqualification.

## Demo round

At the end, per person/team:

1. Maker demos for 3 minutes.
2. Link is opened by colleagues.
3. Game is played for 5 minutes.
4. Group discusses what was played and what they liked.
